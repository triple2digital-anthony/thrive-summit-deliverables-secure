Code Examples (recommended stack + how it complements the platform)

Recommended tech stack (production-ready)

Frontend

  • Next.js (App Router) + TypeScript for fast pages + server actions/routes
  • Tailwind CSS for consistent UI (matches this doc)
  • Vercel/Render static for marketing pages; app can run on Render

Backend + data

  • Postgres (Render Postgres) for leads, tickets, VIP apps, sponsor inventory
  • Prisma for schema + migrations
  • Auth (Clerk or Auth0) for true protected portals (no client-side passwords)
  • HubSpot (or Pipedrive) + webhooks for CRM routing (matches the CRM stages deliverable)

How this complements the Thrive Summit platform: it turns the “documents” above into live flows (lead capture → qualification → ticketing → VIP application → onboarding → reporting) while keeping data HIPAA-safe via aggregated metrics and clear access boundaries.

Example 1 — Lead capture endpoint (Next.js Route Handler)

Use for: lead magnets, QR opt-in, landing page forms (see “Capture + Qualification Flow”).

app/api/leads/route.ts
// app/api/leads/route.ts
import { NextResponse } from "next/server";
import { z } from "zod";
import { prisma } from "@/lib/prisma";

const LeadSchema = z.object({
  email: z.string().email(),
  name: z.string().min(1),
  role: z.enum(["owner", "dentist", "om", "hygienist", "assistant", "other"]).optional(),
  source: z.string().min(1), // e.g. "scorecard", "capacity-calculator", "summit-qr"
});

export async function POST(req: Request) {
  const json = await req.json();
  const lead = LeadSchema.parse(json);

  await prisma.lead.upsert({
    where: { email: lead.email },
    update: { name: lead.name, role: lead.role, source: lead.source },
    create: { ...lead },
  });

  return NextResponse.json({ ok: true });
}

Example 2 — Data model (Prisma) that mirrors your CRM stages

Use for: pipeline stages from “CRM Pipeline + Stages”.

prisma/schema.prisma
// prisma/schema.prisma
model Lead {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  email     String   @unique
  name      String
  role      String?
  source    String

  practice  Practice?
  stage     PracticeStage @default(CAPTURED_LEAD)
  vipApp    VipApplication?
}

model Practice {
  id          String @id @default(cuid())
  leadId      String @unique
  lead        Lead   @relation(fields: [leadId], references: [id])

  name        String?
  city        String?
  state       String?
  operatories Int?
  hygienists  Int?
}

model VipApplication {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())

  leadId    String   @unique
  lead      Lead     @relation(fields: [leadId], references: [id])

  constraint String   // "capacity" | "acceptance" | "protocols" | "accountability"
  weeksBookedOut Int?
}

enum PracticeStage {
  CAPTURED_LEAD
  ENGAGED
  TRIAGE_SCHEDULED
  QUALIFIED_FOR_SUMMIT
  REGISTERED
  ATTENDED_SUMMIT
  VIP_APPLICATION_SUBMITTED
  VIP_FIT_CALL_COMPLETED
  CLOSED_WON_VIP
  DELIVERY
  CONVERTED_TO_FRACTIONAL
  ALUMNI
}

Example 3 — VIP routing rule (capacity vs implementation vs fractional)

Use for: mapping “Qualification Flow” → VIP Track A/B/C and creating tasks in CRM.

lib/routing.ts
// lib/routing.ts
export type Constraint = "capacity" | "acceptance" | "protocols" | "accountability" | "unknown";
export type VipTrack = "TRACK_A_CAPACITY_SPRINT" | "TRACK_B_IMPLEMENTATION" | "TRACK_C_FRACTIONAL" | "THRIVE_TOGETHER";

export function recommendVipTrack(input: {
  constraint: Constraint;
  weeksBookedOut?: number | null;
  wantsOngoingOversight?: boolean;
}): VipTrack {
  if (input.wantsOngoingOversight || input.constraint === "accountability") return "TRACK_C_FRACTIONAL";
  if (input.constraint === "capacity") return "TRACK_A_CAPACITY_SPRINT";
  if (input.constraint === "acceptance" || input.constraint === "protocols") return "TRACK_B_IMPLEMENTATION";
  if ((input.weeksBookedOut ?? 0) >= 6) return "TRACK_A_CAPACITY_SPRINT";
  return "THRIVE_TOGETHER";
}
